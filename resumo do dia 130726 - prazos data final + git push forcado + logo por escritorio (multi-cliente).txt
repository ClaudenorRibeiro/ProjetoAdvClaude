A) PRAZOS — DATA FINAL pode ser digitada direto (bidirecional) e virou OBRIGATÓRIA
   — só LOCAL, SEM SQL, build OK
--------------------------------------------------------------------------------
PEDIDO: ao cadastrar um prazo, poder informar a DATA FINAL direto, sem precisar
preencher a quantidade de dias. E todo prazo passa a EXIGIR uma data final.

O QUE HAVIA ANTES (verificado na fonte): só dava para informar a QUANTIDADE de dias
e o sistema calculava a data final (dias úteis/corridos, com feriados). O campo
"Data final" até existia e era editável na tela, MAS o backend IGNORAVA a data
digitada — se a quantidade ficasse vazia, o prazo era salvo SEM vencimento (nulo).

DECISÃO DO USUÁRIO (confirmada): quando ele digitar a data final, ELA é a que MANDA
— vira o vencimento exato. A quantidade de dias vira só referência.

O QUE FOI FEITO:
- Backend `services/calendarioService.js`: nova função `calcularQuantidade(inicio,
  final, tipo)` — o INVERSO de calcularVencimento. Úteis: conta no `calendario`
  os dias úteis entre início e fim (inclusive os dois). Corridos: diferença + 1.
- Backend `controllers/prazosController.js`: `criar` e `editar` agora leem
  `data_final` do body → é ela que vira `data_vencimento` (se não vier, calcula
  pela quantidade; se nada vier, erro amigável "A data final é obrigatória").
  Novo endpoint `calcularDias` + rota GET `/prazos/calcular-dias`.
- Frontend `pages/Prazos/Prazos.js` (os 2 modais: Novo e Editar, reaproveitados
  também no PastaDetalhe): estado `modo` ('dias' | 'data', começa null) diz qual
  campo o usuário mexeu por último, para os dois cálculos (dias->data e data->dias)
  NÃO ficarem se recalculando em loop. Começar null faz com que, ao ABRIR a edição
  de um prazo, nada seja recalculado (preserva a data e a quantidade salvas).
  Data final virou OBRIGATÓRIA na validação da tela. `services/api.js`: `calcularDias`.
- SEM mudança de banco.

FALTA: usuário testar na tela (digitar só a quantidade; digitar só a data final;

salvar_pc_escrit_no_Git.bat) troquei `git push origin main` por
`git push --force origin main` (com comentário explicativo no arquivo).
- `--force-with-lease` NÃO serve aqui (ele bloquearia justamente quando o remoto
  mudou, que é o caso) -> é `--force` puro, coerente com o "nunca baixar".
- O `Deploy/atualizacao/1-AtualizarSistema.sh` do servidor NÃO precisou mudar: ele
