2) FIX — clique no checkbox do MÓDULO-PAI não expande/oculta mais a barra (LOCAL)
--------------------------------------------------------------------------------
Problema relatado: nas linhas de módulo-PAI das permissões (ex.: "Audiências
(menu)"), clicar num checkbox marcava/desmarcava E ao mesmo tempo abria/fechava
os sub-itens.
Causa: a linha inteira do módulo-pai é clicável (expande/oculta); como os
checkboxes ficam dentro dela, o clique "vazava" para a linha.
Correção: os checkboxes do módulo-pai passaram a "segurar" o clique
(stopPropagation), para não chegar na barra. Agora: clicar no checkbox = só
marca/desmarca; clicar na barra (no nome do módulo) = só mostra/oculta os
sub-itens. Só as linhas-pai foram afetadas.
Mesmo arquivo (Configuracoes.js). Feito DEPOIS do deploy do item 1 =>
NÃO commitado/deployado; está só no LOCAL.

4) RECURSO NOVO — CAMPO NACIONALIDADE (obrigatório, via tabela auxiliar)
--------------------------------------------------------------------------------
Descoberta (verificada na fonte): a nacionalidade NÃO era coletada em lugar
nenhum — não havia campo na tela, o backend não tratava, e não existia coluna no
banco. Só existia como variável de documento, que por isso saía SEMPRE vazia.

Decisões do usuário (uma a uma):
  - Campo NOVO e OBRIGATÓRIO em Pessoa Física.
  - Alimentado por uma TABELA À PARTE (para não ter erro de escrita), igual ao
    padrão do "gênero".
  - Nome da tabela: "nacionalidade" (SEM prefixo tbl — segue as irmãs genero/
    estado_civil/profissao; o prefixo tbl é só para tabelas de processo).
  - "Brasileira" fixada no id = 1.
  - TODOS os registros já existentes de pessoas_fisicas passam a ter
    nacionalidade_id = 1 (backfill) — assim, tornar o campo obrigatório NÃO
    trava a edição da base já importada (as 4.938 pessoas).

Antes de codar, mapeei por INTEIRO o padrão do "gênero" em todo o sistema (banco,
backend e frontend) para replicar sem quebrar nada. Também confirmei que a rotina
"Limpar dados de teste" usa uma lista fixa de tabelas — a nova tabela
"nacionalidade" fica preservada automaticamente (como o gênero).

SQL ENTREGUE (o usuário roda no HeidiSQL — LOCAL). Rodar UMA vez:
  1) CREATE TABLE nacionalidade (id INT AUTO_INCREMENT PK, nome VARCHAR(50) NOT NULL)
     — estrutura idêntica à tabela genero.
  2) INSERT INTO nacionalidade (id, nome) VALUES (1, 'Brasileira').
  3) ALTER TABLE pessoas_fisicas ADD COLUMN nacionalidade_id INT NULL AFTER
     genero_id + KEY + CONSTRAINT pessoas_fisicas_ibfk_4 FOREIGN KEY
     (nacionalidade_id) REFERENCES nacionalidade(id).
  4) UPDATE pessoas_fisicas SET nacionalidade_id = 1 (backfill — todos Brasileira).
  Observações: os comandos CREATE/ALTER do MySQL não voltam atrás numa transação
  (cada um se confirma sozinho), por isso é um script de estrutura, roda 1x; a
  ordem importa (a Brasileira id 1 tem que existir antes do backfill, senão a
  chave estrangeira barra). A coluna é NULLABLE no banco (igual genero_id) — a
  obrigatoriedade é validada só na tela. NÃO foi registrado na tabela
  controle_versao_banco (o usuário dispensou nesta alteração específica).
